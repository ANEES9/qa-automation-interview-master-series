# Q034. What are records in Java?

## Interview Answer

A record is a compact Java class used to create immutable data carriers.

Records automatically provide a constructor, accessor methods, `equals()`, `hashCode()`, and `toString()`. In automation, records are useful for test data models, API response models, and expected UI data.

---

## Detailed Explanation

Records became a standard Java feature in Java 16 and are fully relevant in Java 21.

Example:

```java
public record User(String username, String role) {
}
```

This automatically creates:

- A constructor.
- Accessors like `username()` and `role()`.
- `equals()`.
- `hashCode()`.
- `toString()`.

Records are shallowly immutable. The record fields cannot be reassigned, but mutable objects inside a record can still be modified.

| Feature | Class | Record |
| --- | --- | --- |
| Boilerplate | More | Less |
| Data carrier | Manual | Built-in |
| Fields | Flexible | Final components |
| Accessors | Getters by convention | Component methods |
| Best use | Behavior-rich objects | Simple immutable data |

---

## Why Interviewers Ask This

Interviewers ask this to check awareness of modern Java and clean data modeling.

For automation engineers, records are useful when representing expected table rows, API response data, test users, and configuration objects.

---

## Real Project Example

In an HR automation project, an employee row from the UI can be represented as an `EmployeeRecord`.

The expected employee data from an API can then be compared with the actual UI data using the automatically generated `equals()` method.

---

## Java Code Example

```java
public record EmployeeRecord(String employeeId, String fullName, String department) {
    public EmployeeRecord {
        if (employeeId == null || employeeId.isBlank()) {
            throw new IllegalArgumentException("Employee ID must not be blank");
        }
        if (fullName == null || fullName.isBlank()) {
            throw new IllegalArgumentException("Full name must not be blank");
        }
    }
}
```

---

## Code Explanation

- `EmployeeRecord` is an immutable data carrier.
- The compact constructor validates required fields.
- Java automatically provides accessors like `employeeId()`.
- Java also generates `equals()`, `hashCode()`, and `toString()`.

---

## Best Practices

- Use records for simple immutable data models.
- Validate important data in compact constructors.
- Avoid putting heavy business logic inside records.
- Be careful with mutable components like `List`.
- Use records for expected and actual data comparison in tests.

---

## Common Mistakes

- Thinking records are only DTOs and cannot have validation.
- Expecting JavaBean getters like `getEmployeeId()`.
- Using records for highly mutable objects.
- Forgetting that nested mutable objects can still change.
- Treating records as a replacement for every class.
